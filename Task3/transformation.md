# Точки трансформации RetailCore

### Точка 1: Нормализация статусной модели заказа (первая очередь)

**Проблема:**
Логика переходов статусов разбросана по монолиту, batch-процедурам и ручным интерфейсам. Клиентские и внутренние статусы расходятся. Поддержка не имеет единой картины. Статусы не подходят как для клиентов, так и для технических операторов

**Почему эта точка подходит для изменений:**
- Не затрагивает checkout (чтение статусов отделено от оформления)
- Прямой эффект для бизнеса - снижение времени обработки обращений, улучшение клиентского опыта
- зона проще поддается изменениям (более безопасная) по сравнению с чекаутом и прайсингом

**Зависимости и риски:**
- Риск: (изменение логики статусов может повлиять на batch-процессы и интеграции)
- Риск: рассинхрон старой и новой статусной модели на время периходного периода

### Точка 2: Унификация интеграционного слоя (Integration Gateway) (первая очередь)

**Проблема:**
Интеграции с внешними системами реализованы разнородно: синхронные REST, обмен через staging-таблицы в БД, очереди с разной семантикой, csv. Отсутствуют единые контракты, идемпотентность и гарантии доставки.

**Почему эта точка подходит для изменений:**
- Снижает связность между монолитом и внешними/вспомогательными системами
- Снижает последующую сложность миграции БД
- Дает основу для дальнейшего выделения логистических и платежных сервисов

**Зависимости и риски:**
- Зависимость - потенциально интеграции со внешними партенрами могут повлечь пересмотр механизма интеграции/контрактов/протоколов
- Риск - изменение протоколов интеграций может повлечь недоступность системы

### Точка 3: Выделение аналитического контура из операционной БД (первая очередь)

**Проблема:**
Аналитические запросы выполняются напрямую к БД Oracle, создавая нагрузку и влияя на доступность основного функционала. Разные витрины дают противоречивые цифры.

**Почему эта точка подходит для изменений:**
- Не затрагивает операционные процессы оформления заказа
- Снижает нагрузку на Oracle что критично для прохождения сезона

**Зависимости и риски:**
- Риск - требует подготовки инфраструктуры для аналитического хранилища

### Точка 4: Выделение Pricing & Promo Logic (вторая очередь)

**Проблема:**
Несколько механизмов расчета цены и скидок. Разные правила для разных каналов. Неявная Fallback-логика. Изменения правил затрагивают множество компонентов.

**Почему эта точка подходит для изменений:**
- Высокая бизнес-ценность: прямо влияет на скорость запуска новых промо-механик

**Зависимости и риски:**
- Зависимость: требует Точек 1-3 для безопасного выделения, **не имеет смысла в реализации в первую очередь**
- Риск - критическое влияние возможных ошибок в расчете цены

### Точка 5: Выделение Delivery Routing (вторая очередь)

**Проблема:**
Логика выбора маршрута доставки распределена по коду, конфигурациям интеграционного слоя и SQL-скриптам. Ручные настройки приоритетов. Разнородные протоколы связи с партнерами.

**Почему эта точка подходит для изменений:**
- Улучшение критичной части опыта пользователя - обещания по доставке

**Зависимости и риски:**
- Зависимость: требует Точки 2, **не имеет смысла в реализации в первую очередь**
- Риск - ошибка маршрутизации влияет на сроки доставки и клиентский опыт
- Риск - множество внешних партнерских API с разным поведением, разной степенью готовности поддержки иных механизмов интеграции

### Точка 6: Нормализация Checkout Core в рамках монолита (третья очередь)

**Проблема:**
Длинные транзакции, синхронные вызовы внешних систем в рамках checkout, смешение ответственности.

**Почему эта точка подходит для изменений:**
- Критический компонент пользовательского опыта влияющий на конверсию и ретеншен пользователя

**Зависимости и риски:**
- Checkout — самая рискованная зона, прямые изменения опасны, изменения должны быть покрыты метриками и проводится постепенно. **Не предполагается выделять Checkout из монолита ввиду ограничений сроков и отсутствия понимания всех зависимостей**

## 2. Целевая архитектура (TO-BE) на горизонте 12 месяцев

Монолит сохраняется как ядро системы, его зона ответственности постепенно снижается. Из монолита выделяются сервисы для статусов, аналитики, интеграций и ценообразования.

- Монолит сохраняется как оркестратор checkout по причине рискованности выделения и высокого влияния на метрики компании
- Выделенны сервисы для статусов, аналитики, интеграций
- Выделен интеграционный шлюз для интеграции внешних систем вместо обмена через БД
- Выделено отдельное аналитическое хранилище в отрыве от продуктовой БД что благоприятно влияет на нагрузку и стабильность системы в пики
- Сохранение единой БД на период ограничений срока реализации целевого решения
