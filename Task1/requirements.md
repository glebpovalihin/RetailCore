# Требования к платформе RetailCore

## 1. Ключевые ФТ

|ID| Требование |
|---|-----------|
|1| Платформа должна поддерживать базовый процесс оформления заказа: выбор товаров, расчет стоимости, применение скидок/промо, выбор доставки и оплаты |
|2| Платформа должна управлять жизненным циклом заказа (статусы: new, confirmed, reserved, paid, shipped, delivered, cancelled) |
|3| Платформа должна предоставлять клиентам информацию о статусе заказа через каналы продаж |
|4 | Платформа должна обеспечивать прозрачность статусов заказа как для клиентов, так и для внутренних пользователей (поддержка, операции) |
|5| Платформа должна обеспечивать пост-обработку заказа: проверка данных, резервирование товаров, вызовы внешних систем, платежные и логистические сценарии|
|6| Платформа должна поддерживать несколько каналов продаж: интернет-магазин, мобильное приложение, маркетплейсы с разной логикой обработки |
|7| Платформа должна интегрироваться с внешними платежными провайдерами для создания платежных сессий и обработки возвратов |
|8| Платформа должна интегрироваться со службами доставки для резервирования слотов и маршрутизации |
|9| Платформа должна формировать аналитические отчеты для бизнес-пользователей (маркетинг, финансы, операции) |
|10| Платформа должна поддерживать партнерские интеграции (маркетплейсы, логистические партнеры) с различными протоколами взаимодействия |
|11| Платформа должна обеспечивать административные инструменты для ручной обработки исключительных ситуаций и поддержки |

## 2. Архитектурно значимые НФТ

| ID | Требование |
|----|-----------|
| 1 | Трансформация должна выполняться поэтапно, без остановки бизнес-процессов (нулевое окно недоступности для критичных операций) |
|2 | Каждый этап трансформации должен обеспечивать возможность отката |
| 3 | Изменения не должны ухудшать текущую конверсию заказа, стабильность оплаты и выполнение обещаний по доставке |
|4 | Система должна обеспечивать наблюдаемость: прослеживаемость жизненного цикла заказа, история изменений, корреляция событий |
|5| Система должна снижать зависимость от ручных операций |
|6| Архитектура должна поддерживать постепенный отказ от Oracle и других зарубежных компонентов (импортонезависимость) |
|7| Платформа должна обеспечивать возможность скалирования ресурсов для обеспечения роста нагрузки и быть готовой к пиковым периодам (распродажи, сезонные пики) |
| 8 | Решение должно обеспечивать сокращение времени от бизнес-идеи до внедрения для типовых коммерческих изменений |

## 3. Ограничения

| ID | Ограничение |
|----|------------|
| 1 | Нельзя останавливать бизнес-процессы на время миграции — нулевое окно недоступности для оформления заказов |
| 2 | Неприемлем подход при котором система переписывается с нуля одномоментно |
| 3 | Каждый этап трансформации должен приносить измеримую бизнес-ценность |
| 4 | Наиболее рискованная зона (checkout, оплата, обещания по доставке) требует особой осторожности и проработки возможности отката |
| 5 | Через три месяца нужна полная картина текущего состояния платформы |
| 6 | Через год ожидается запуск новых сервисов (персонализация, гибкое управление предложениями, партнерские сценарии, экосистема) |
| 7 | Документация неполная, часть знаний сосредоточена у конкретных сотрудников |

## 4. Допущения

| ID | Допущение | Обоснование |
|----|----------|-------------|
| 1 | Бизнес-требования к новым сервисам (персонализация, экосистема) будут уточнены позже и не блокируют начало архитектурной трансформации | Дмитрий: «сначала архитектурное понимание, потом целевое состояние, потом программа перехода» |
| 2 | Возможно параллельное сосуществование старого и нового контуров в течение переходного периода (strangler fig pattern) |В рамках интервью не было ограничений для подобных подходов|
