# Architecture Communication Canvas

## Ценностное обоснование

Цель: платформа RetailCore перестала обеспечивать нужную бизнесу скорость изменений. Каждый новый коммерческий сценарий требует непредсказуемых усилий, затрагивает множество компонентов и несет повышенный риск. Система критична для выручки, но ее текущее устройство ограничивает стратегию компании.

### Ожидаемый результат для бизнеса:

- Типовые коммерческие изменения запускаются быстрее и с предсказуемыми сроками и стоимостью
- Меньше инцидентов и ручных операций в критичных сценариях заказа
- Единая доверенная картина данных по заказам
- Меньше ситуаций, когда одна задача требует одновременного участия нескольких команд только потому, что изменение пересекает старые технические границы

### Ожидаемый результат для пользователей (клиентов и поддержки):

- Клиент видит понятный и согласованный статус заказа во всех каналах
- Оператор поддержки получает целостную картину заказа в одном интерфейсе, без необходимости обращаться в смежные команды для ответа клиенту
- При отклонении от стандартного сценария система объясняет, что произошло и почему

## Ключевые заинтересованные стороны

| Сторона | Цель |
|---------|------------------------|
| Клиенты (покупатели) | Быстрое оформление, корректная цена и срок, понятный статус, предсказуемое выполнение заказа |
| Операторы поддержки | Единая карточка заказа с полной traced-историей; понятные причины отклонений; меньше ручных переключений между системами |
| Director of Digital Commerce | Ускорение коммерческих изменений; стабильность критичных сценариев; прозрачность данных; снижение ручных операций |
| Бизнес-подразделения (маркетинг, финансы, операции) | Единая доверенная картина данных; отсутствие конфликта цифр между витринами |
| Head of Customer Support | Прозрачность статусов; снижение времени обработки обращений; меньше межкомандных эскалаций |

## Ключевые функции

**Основные сценарии:**

| Функция | Описание поведения системы |
|---------|---------------------------|
| Принять и оформить заказ | Получить состав заказа от клиента, рассчитать итоговую цену со скидками, предложить доступные способы доставки и оплаты, зафиксировать заказ |
| Подтвердить выполнимость заказа | Проверить наличие товара, зарезервировать слот доставки, создать платежную сессию |
| Показать клиенту актуальное состояние заказа | Отобразить текущий статус, срок доставки и историю изменений в понятной форме во всех каналах |
| Уведомить клиента об изменении | Сообщить, если изменился срок, состав, способ доставки или статус — с объяснением причины |
| Передать заказ внешнему партнеру | Отправить данные заказа службе доставки, маркетплейсу или платежному провайдеру; получить и обработать подтверждение |
| Обработать отклонение от стандартного сценария | При сбое доставки, оплаты или резервирования — перевести заказ в понятное исключительное состояние с указанием причины и дальнейших процессов |
| Предоставить оператору поддержки полную картину заказа | Показать в одном интерфейсе: состав, статусы, историю изменений, причины отклонений, взаимодействия с внешними системами — без необходимости обращаться к смежным командам |
| Предоставить аналитическую отчетность | Сформировать согласованную картину по заказам, выручке, кампаниям для маркетинга, финансов и операций — без расхождений между витринами |
| Предоставить бизнесу инстурмен настройки коммерческих правил | Позволить изменить условие промо, правило доставки или партнерское предложение без изменения кода платформы |

## Требования к качеству

| Атрибут (ISO 25010) | Требование | Как подтвердить |
|---------------------|-----------|----------------|
| Надежность (Reliability)| При сбое внешней системы (доставка, платеж) заказ не теряется, а переходит в контролируемое исключительное состояние | Наличие тестового платежного шлюза с возможностью проведения тестового заказа |
| Доступность (Availability) | Оформление заказа доступно всегда; плановые и внеплановые работы не останавливают checkout | За 12 месяцев 0 минут простоя checkout по причине архитектурных изменений |
| Временные характеристики (Time behaviour) | Клиент видит статус заказа с задержкой не более допустимой | — ограничение не задано, требует обсуждения и заполнения пробела |
| Масштабируемость (Scalability) | Пиковая нагрузка сезонных распродаж не вызывает деградации времени оформления и отображения статусов | — ограничение не задано, требует обсуждения и заполнения пробела |
| Наблюдаемость (Observability) | По любому заказу можно восстановить полную цепочку событий от оформления до текущего состояния | Оператор поддержки видит traced-историю заказа в одном интерфейсе |
| Понятность (Appropriateness recognisability) | Клиент понимает текущий статус заказа без дополнительных обращений в поддержку | Количество обращений в поддержку снижено относительно baseline |
| Защищенность (Security) | Платежные данные обрабатываются в соответствии с PCI DSS | — ограничение не задано, требует обсуждения и заполнения пробела |
| Модифицируемость (Modifiability) | Типовое коммерческое изменение требует изменения конфигурации, а не кода | Добавление нового правила скидки выполнено изменением конфигурации менее чем за 1 рабочий день |
| Восстанавливаемость (Recoverability) | Каждый этап трансформации имеет проверенный сценарий отката | Системные тесты работы с релизами |
| Совместимость (Compatibility) | Внешние API для маркетплейсов и каналов продаж сохраняют обратную совместимость | Существующие интеграции работают без изменений на стороне партнера |

## Деловой контекст
- Пользователи каналов продаж генерируют нагрузку; определяют требования к UX статусов
- Внешние источники заказов ограничивают изменения API; их статусные модели могут расходиться с внутренней
- Внешние обработчики платежей определяют доступность checkout
- Службы доставки - задержки порождают исключеительные сценарии

**Граница текущего объема:** инициатива не включает полную замену монолита.

## Ключевые решения

### Унаследованные решения, определяющие текущее состояние

- Нельзя заменить монолит одномоментно; только поэтапное выделение через Strangler Fig
- Полная миграция хранилища — долгосрочная цель; на первых этапах только снижение зависимости (вынос логики, отделение аналитики)
- Сокращение границ транзакций в которых выполняются синхронные вызовы внешних систем; circuit breaker; асинхронная пост-обработка
- Автоматизация исключений; контролируемые ручные сценарии на переходный период и их планомерное снижение

### Сознательные отказы
- Полная замена монолита - не реалистична так как требует остановик бизнеса, стейкхолдеры отвергают данный подход
- Единомоментная миграция с Oracle чревата высоким риском (большой объем бд, высокая связность данных)
- Единомоментный переход к микросервисной архитектуре не оправдан ввиду конфликта с требованием управляемой эволюции с измеримой ценностью каждого шага

## Составные части системы

- Каналы продаж - Интернет-магазин и мобильное приложение для приема заказов от клиентов
- Монолитное ядро обработки заказов - Сохраняет checkout и оркестрацию, но с улучшенной внутренней структурой: сокращенные транзакции, circuit breaker, вызовы к новым сервисам вместо прямых SQL в БД
- Сервис управления статусами - Единый источник правды для статусов заказа; два представления: клиентское (абстрактное) и внутреннее (детальное); traced-история изменений с причинами
- Сервис расчета цены и скидок - Унифицированный расчет для всех каналов; конфигурируемые правила вместо хардкода
- Интеграционный шлюз - Единая точка взаимодействия с внешними системами; стандартные протоколы с гарантированной доставкой и идемпотентностью
- Поток событий - Единый источник событий заказа для аналитики, интеграций и межсервисного взаимодействия
- Аналитическое хранилище - Выделенное хранилище для отчетов и витрин
- Операционное хранилище (Oracle) - Сохраняется для операционных данных заказа
- Административные инструменты - Ручная проверка и корректировка заказов операторами поддержки

## Риски и недостающая информация

- Скрытые зависимости в Oracle - потенциальный фон ошибок
- Не известны RTO/RPO для checkout, статусов и аналитики
- Не зафиксированы целевые значения пиковой нагрузки (RPS) для checkout
